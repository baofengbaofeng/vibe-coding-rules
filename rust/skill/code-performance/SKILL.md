## 用户输入

```
$ARGUMENTS
```

## 目标

执行Rust项目代码性能检查，包括：
1. 数据库性能优化检查（查询优化、索引使用、连接池）
2. 缓存策略检查（Redis使用、缓存击穿/雪崩防护）
3. 并发与异步性能检查（Tokio任务调度、锁竞争）
4. 内存管理检查（内存泄漏、过度分配）
5. 算法复杂度检查（时间复杂度、空间复杂度）
6. 输出详细的性能检查报告

## 执行步骤

### 1. 确认项目环境

首先确认当前目录是Rust项目根目录，检查以下技术栈：
- Rust 1.75+
- Tokio 1.35+ 异步运行时
- SQLx 0.7+ 或 SeaORM 数据库访问
- Redis 客户端（可选）
- 性能监控工具（tracing, metrics）

### 2. 数据库性能检查

#### 查询优化检查
- 检查是否使用N+1查询问题
- 检查是否使用JOIN替代多次查询
- 检查是否使用SELECT *（应选择具体字段）
- 检查分页查询实现是否高效

#### 索引使用检查
- 检查高频查询字段是否建立索引
- 检查复合索引使用是否合理
- 检查索引覆盖查询使用

#### 连接池检查
- 检查数据库连接池配置是否合理
- 检查连接泄漏问题
- 检查连接超时配置

### 3. 缓存策略检查

#### Redis缓存检查
- 检查缓存键设计是否合理
- 检查缓存过期策略
- 检查缓存穿透/击穿/雪崩防护

#### 本地缓存检查
- 检查内存缓存使用是否合理
- 检查缓存淘汰策略
- 检查缓存一致性

### 4. 并发与异步性能检查

#### Tokio任务调度检查
- 检查任务拆分是否合理（避免阻塞运行时）
- 检查spawn使用是否恰当
- 检查async/await使用模式

#### 锁竞争检查
- 检查锁粒度是否过粗
- 检查死锁风险
- 检查无锁数据结构使用

### 5. 内存管理检查

#### 内存泄漏检查
- 检查循环引用（Arc + RefCell）
- 检查全局变量内存增长
- 检查未释放资源

#### 内存分配检查
- 检查频繁分配/释放
- 检查大对象分配
- 检查内存碎片

### 6. 算法复杂度检查

#### 时间复杂度检查
- 检查嵌套循环复杂度
- 检查递归算法深度
- 检查数据结构和算法选择

#### 空间复杂度检查
- 检查内存使用是否合理
- 检查临时对象创建
- 检查数据拷贝

## 检查规则参考

### 数据库性能规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| N+1查询 | 使用JOIN或批量查询 | CRITICAL |
| SELECT * | 选择具体字段 | HIGH |
| 连接池 | 合理配置连接数 | MEDIUM |
| 索引 | 高频查询字段建立索引 | HIGH |

### 缓存性能规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| 缓存键设计 | 合理设计避免冲突 | MEDIUM |
| 缓存过期 | 合理设置过期时间 | MEDIUM |
| 缓存穿透 | 使用布隆过滤器或空值缓存 | HIGH |
| 缓存击穿 | 使用互斥锁或永不过期 | HIGH |

### 并发性能规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| 任务拆分 | 合理拆分避免阻塞 | HIGH |
| 锁粒度 | 使用细粒度锁 | HIGH |
| 死锁 | 避免循环等待 | CRITICAL |
| 无锁结构 | 使用并发数据结构 | MEDIUM |

## 注意事项

1. 性能检查需要结合具体业务场景
2. 性能优化要考虑权衡（内存 vs CPU）
3. 建议定期执行性能检查（每月/每季度）
4. 新功能开发必须通过性能检查
5. 遵循AGENTS.md中的性能优先原则
6. 使用criterion进行基准测试
