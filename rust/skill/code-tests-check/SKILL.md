## 用户输入

```
$ARGUMENTS
```

## 目标

执行Rust项目测试检查，包括：
1. 单元测试完整性检查（测试用例覆盖、边界条件）
2. 集成测试检查（API测试、数据库测试）
3. 测试覆盖率检查（行覆盖率、分支覆盖率）
4. 测试质量检查（测试命名、测试结构）
5. 输出详细的测试检查报告

## 执行步骤

### 1. 确认项目环境

首先确认当前目录是Rust项目根目录，检查以下文件是否存在：
- `Cargo.toml`
- `src/` 目录结构
- `tests/` 目录（集成测试）
- 测试依赖配置（mockall, tokio-test等）

### 2. 执行测试运行检查

运行以下命令执行测试：

```
# 运行所有测试
cargo test --verbose

# 运行单元测试
cargo test --lib

# 运行集成测试
cargo test --test "*"
```

捕获输出结果，分析测试通过率。

### 3. 测试覆盖率检查

运行以下命令检查测试覆盖率：

```
# 使用tarpaulin检查覆盖率
cargo tarpaulin --ignore-tests --out Xml

# 或使用grcov
cargo test --coverage
```

分析覆盖率报告，检查是否达到标准：
- 领域层（domain）≥ 90%
- 整体覆盖率 ≥ 85%

### 4. 单元测试完整性检查

根据 `rules/单元测试规范.md` 定义的规范，检查以下内容：

#### 测试结构检查
- 测试模块组织是否合理（#[cfg(test)]模块）
- 测试函数命名是否符合规范（#[test]函数）
- 测试文件位置是否正确（与源码同目录或tests/目录）

#### 测试用例检查
- 是否覆盖正常路径（happy path）
- 是否覆盖错误路径（error cases）
- 是否覆盖边界条件（edge cases）
- 是否包含参数化测试（如果需要）

#### 测试质量检查
- 测试是否独立（不依赖外部状态）
- 测试是否可重复（每次运行结果一致）
- 测试是否快速（执行时间合理）
- 测试断言是否明确（assert_eq!, assert!）

### 5. 集成测试检查

#### API测试检查
- API端点是否都有测试
- 请求/响应验证是否完整
- 认证/授权测试是否覆盖
- 错误状态码测试

#### 数据库测试检查
- 是否使用测试数据库
- 事务管理是否正确
- 数据清理是否完整
- 并发测试是否覆盖

### 6. 测试工具使用检查

#### Mocking检查
- 是否使用mockall进行依赖模拟
- Mock配置是否合理
- 期望设置是否完整

#### 异步测试检查
- 是否使用tokio::test属性
- 异步测试超时设置
- 并发测试场景

## 检查规则参考

### 测试覆盖率标准

| 层级 | 行覆盖率要求 | 说明 |
|------|--------------|------|
| 领域层（domain） | ≥ 90% | 核心业务逻辑必须高覆盖 |
| 应用层（application） | ≥ 85% | 业务服务层需要良好覆盖 |
| API层（api） | ≥ 85% | 接口层需要完整测试 |
| 基础设施层（infrastructure） | ≥ 80% | 外部依赖适配层 |
| 整体项目 | ≥ 85% | 项目整体质量标准 |

### 测试命名规范

| 测试类型 | 命名模式 | 示例 |
|----------|----------|------|
| 单元测试 | test_[功能]_[场景] | test_user_create_success |
| 错误测试 | test_[功能]_[错误类型] | test_user_create_invalid_email |
| 边界测试 | test_[功能]_[边界条件] | test_user_create_empty_name |
| 集成测试 | [模块]_[功能]_test | user_registration_test |

### 测试结构规范

| 要求 | 说明 | 检查级别 |
|------|------|----------|
| 测试模块 | 使用#[cfg(test)]模块 | ERROR |
| 测试函数 | 使用#[test]或#[tokio::test] | ERROR |
| 测试文件 | 与源码同目录或tests/目录 | WARN |
| 测试断言 | 使用assert_eq!, assert!等 | ERROR |
| 测试清理 | 测试后清理状态 | WARN |

## 使用示例

```
# 全项目测试检查
/code-tests-check

# 检查领域层测试
/code-tests-check domain

# 检查API测试
/code-tests-check api
```

## 注意事项

1. 测试检查需要结合具体业务场景
2. 测试覆盖率不是唯一目标，测试质量更重要
3. 建议在代码提交前执行测试检查
4. 新功能开发必须包含相应测试
5. 遵循AGENTS.md中的测试驱动原则
6. 定期运行测试并监控覆盖率趋势
