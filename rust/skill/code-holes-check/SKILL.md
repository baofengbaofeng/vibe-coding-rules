## 用户输入

```
$ARGUMENTS
```

## 目标

执行Rust项目代码安全检查，包括：
1. 注入防护检查（SQL注入、命令注入、路径遍历）
2. 敏感信息检查（硬编码密钥、配置泄露、日志脱敏）
3. 访问控制检查（API鉴权、权限校验、CORS配置）
4. 输入验证检查（参数验证、文件上传验证）
5. 输出详细的安全检查报告

## 执行步骤

### 1. 确认项目环境

首先确认当前目录是Rust项目根目录，检查以下技术栈：
- Rust 1.75+
- Axum 0.7+ 或 Tide/Actix-web
- SQLx 0.7+ 或 SeaORM
- Tokio 1.35+
- Serde 1.0+

### 2. 注入防护检查

#### SQL注入检查
- 检查SQL语句是否使用字符串拼接
- 检查是否使用SQLx参数绑定或SeaORM
- 检查动态表名/列名是否进行白名单校验

#### 命令注入检查
- 检查是否使用 `std::process::Command` 执行用户输入
- 检查命令执行是否使用参数列表而非字符串拼接

#### 路径遍历检查
- 检查文件操作是否验证路径安全性
- 检查是否使用规范化路径函数

### 3. 敏感信息检查

#### 硬编码密钥检查
- 检查代码中是否硬编码数据库密码、JWT密钥、API密钥等
- 检查是否使用环境变量或加密存储敏感配置

#### 配置泄露检查
- 检查 `.env` 文件是否包含敏感信息
- 检查配置文件是否提交到版本控制
- 检查Dockerfile是否包含敏感信息

#### 日志脱敏检查
- 检查日志输出是否包含敏感信息（密码、令牌、个人信息）
- 检查敏感字段是否进行脱敏处理

### 4. 访问控制检查

#### API鉴权检查
- 检查所有API是否经过认证中间件验证
- 检查是否有新增的无鉴权接口
- 检查认证中间件是否正确实现

#### 权限校验检查
- 检查业务层是否进行数据权限校验
- 检查是否可能发生越权访问
- 检查角色和权限管理是否完整

#### CORS配置检查
- 检查CORS是否使用白名单而非允许所有来源
- 检查CORS中间件是否正确配置

### 5. 输入验证检查

#### 参数验证检查
- 检查是否使用validator库进行输入验证
- 检查验证规则是否完整（长度、格式、范围等）
- 检查是否验证所有用户输入

#### 文件上传验证检查
- 检查文件上传是否验证类型和大小
- 检查是否使用白名单限制文件类型
- 检查文件存储路径是否安全

### 6. 安全配置检查

#### 安全响应头检查
- 检查是否设置安全相关的HTTP响应头
- 检查生产环境是否强制HTTPS

#### 速率限制检查
- 检查是否配置速率限制防止暴力破解
- 检查分布式限流实现

## 检查规则参考

### 注入防护规则

| 检查项 | 安全要求 | 风险等级 |
|--------|----------|----------|
| SQL注入 | 使用参数绑定，禁止字符串拼接 | CRITICAL |
| 命令注入 | 使用参数列表，禁止shell拼接 | HIGH |
| 路径遍历 | 规范化路径，检查目录边界 | MEDIUM |

### 敏感信息规则

| 检查项 | 安全要求 | 风险等级 |
|--------|----------|----------|
| 硬编码密钥 | 使用环境变量或加密存储 | HIGH |
| 配置泄露 | 敏感配置不提交版本控制 | MEDIUM |
| 日志脱敏 | 敏感字段脱敏后输出 | MEDIUM |

### 访问控制规则

| 检查项 | 安全要求 | 风险等级 |
|--------|----------|----------|
| API鉴权 | 所有API必须经过认证 | HIGH |
| 权限校验 | 业务层数据权限校验 | HIGH |
| CORS配置 | 使用白名单，禁止所有来源 | MEDIUM |

## 使用示例

```
# 全项目安全检查
/code-holes-check

# 检查认证相关模块
/code-holes-check auth

# 检查指定API模块
/code-holes-check src/api
```

## 注意事项

1. 安全检查需要结合具体业务场景
2. 安全风险等级根据业务影响评估
3. 建议定期执行安全检查（每周/每月）
4. 新功能开发必须通过安全检查
5. 遵循AGENTS.md中的安全优先原则
6. 运行cargo audit检查依赖漏洞
