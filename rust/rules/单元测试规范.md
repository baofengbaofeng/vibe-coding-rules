# 单元测试规范

## 概述
本文档定义Rust项目的单元测试编写规范，包括测试结构、命名约定、Mock使用、覆盖率要求等，确保测试代码的质量和可维护性。

---

## 测试原则

### 1. FIRST原则
- **Fast**: 测试应该快速执行
- **Independent**: 测试应该相互独立
- **Repeatable**: 测试应该可重复执行
- **Self-validating**: 测试应该自我验证
- **Timely**: 测试应该及时编写

### 2. 测试金字塔
- **单元测试**: 70% - 测试单个函数或模块
- **集成测试**: 20% - 测试模块间集成
- **端到端测试**: 10% - 测试完整业务流程

### 3. 测试驱动开发（TDD）
- 红-绿-重构循环
- 先写测试，再写实现
- 测试即文档

## 断言规范

### 1. 基本断言
```
use assert_matches::assert_matches;

#[test]
fn test_basic_assertions() {
    let result = add(2, 3);
    
    // 相等断言
    assert_eq!(result, 5);
    assert_ne!(result, 0);
    
    // 布尔断言
    assert!(result > 0);
    assert!(!result.is_negative());
    
    // 模式匹配断言
    let value: Result<i32, String> = Ok(42);
    assert_matches!(value, Ok(42));
    
    // 错误断言
    let err_result: Result<i32, String> = Err("error".to_string());
    assert!(err_result.is_err());
    
    // 包含断言
    let message = "Hello, World!";
    assert!(message.contains("World"));
    
    // 近似相等（浮点数）
    let pi = 3.14159265359;
    assert!((pi - 3.14).abs() < 0.01);
}
```

## Mock使用规范

### 1. Mock对象定义
```
use mockall::automock;

// 定义需要Mock的trait
#[automock]
pub trait UserRepository {
    fn find_by_id(&self, id: i64) -> Result<Option<User>, Error>;
    fn save(&self, user: &User) -> Result<(), Error>;
    fn delete(&self, id: i64) -> Result<(), Error>;
}

// 在测试中使用Mock
#[test]
fn test_with_mock() {
    // 创建Mock对象
    let mut mock_repo = MockUserRepository::new();
    
    // 设置期望行为
    mock_repo.expect_find_by_id()
        .with(predicate::eq(1))
        .times(1)
        .returning(|_| Ok(Some(User::new(1, "testuser"))));
    
    mock_repo.expect_save()
        .withf(|user| user.id == 1)
        .times(1)
        .returning(|_| Ok(()));
    
    // 使用Mock对象进行测试
    let service = UserService::new(Arc::new(mock_repo));
    let result = service.update_user(1, "newname");
    
    assert!(result.is_ok());
}
```

### 2. Mock配置
```
#[test]
fn test_mock_configuration() {
    let mut mock = MockService::new();
    
    // 设置调用次数
    mock.expect_process()
        .times(1..3)  // 调用1到3次
        .returning(|_| Ok(()));
    
    // 设置调用顺序
    mock.expect_step1()
        .times(1)
        .returning(|| Ok(()));
    
    mock.expect_step2()
        .times(1)
        .after(step1)  // 在step1之后调用
        .returning(|| Ok(()));
    
    // 设置不同参数的不同返回值
    mock.expect_get_value()
        .with(predicate::eq(1))
        .returning(|_| 100);
    
    mock.expect_get_value()
        .with(predicate::eq(2))
        .returning(|_| 200);
    
    // 设置默认返回值
    mock.expect_get_value()
        .returning(|id| id * 10);
}
```

```
fn benchmark_algorithm(c: &mut Criterion) {

}
```

```
#[tokio::test]
async fn test_concurrent_load() {
    use tokio::task;
    
    let app = create_test_app();
    let client = TestClient::new(app);
    
    let start_time = std::time::Instant::now();
    let mut tasks = Vec::new();
    
    // 模拟100个并发用户
    for i in 0..100 {
        let client = client.clone();
        let task = task::spawn(async move {
            for _ in 0..10 {  // 每个用户执行10次请求
                let response = client.get(&format!("/api/users/{}", i)).await;
                assert!(response.status().is_success());
            }
        });
        tasks.push(task);
    }
    
    // 等待所有任务完成
    for task in tasks {
        task.await.unwrap();
    }
    
    let duration = start_time.elapsed();
    
    // 性能断言：1000个请求应该在5秒内完成
    assert!(
        duration < std::time::Duration::from_secs(5),
        "1000个请求耗时{:?}，超过5秒限制",
        duration
    );
}
```

```
#[tokio::test]
async fn test_stress_conditions() {
    let app = create_test_app();
    let client = TestClient::new(app);
    
    // 测试内存使用
    let mut handles = Vec::new();
    
    for i in 0..1000 {
        let client = client.clone();
        let handle = tokio::spawn(async move {
            let response = client.get(&format!("/api/users/{}", i)).await;
            response.status().is_success()

    // 检查是否有内存泄漏
    let success_count = futures::future::join_all(handles)
        .await
        .into_iter()
        .filter(|r| r.as_ref().unwrap_or(&false))
        .count();
    
    assert!(success_count > 900, "成功率应该超过90%");
}
```

```
// tests/helpers/mod.rs
pub mod test_utils {
    use std::sync::Once;
    
    static INIT: Once = Once::new();
    
    /// 初始化测试环境（只执行一次）
    pub fn setup_test_environment() {
        INIT.call_once(|| {
            // 设置日志
            tracing_subscriber::fmt()
                .with_test_writer()
                .init();
            
            // 设置随机种子
            let seed = std::time::SystemTime::now()
                .duration_since(std::time::UNIX_EPOCH)
                .unwrap()
                .as_secs();
            rand::seed_from_u64(seed);
        });
    }
    
    /// 创建测试数据库连接池
    pub async fn create_test_pool() -> PgPool {
        let database_url = std::env::var("TEST_DATABASE_URL")
            .unwrap_or_else(|_| "postgresql://test:test@localhost/test".to_string());
        
        PgPoolOptions::new()
            .max_connections(5)
            .connect(&database_url)
            .await
            .unwrap()
    }
    
    /// 生成测试数据
    pub fn generate_test_user(id: i64) -> User {
        User {
            id,
            username: format!("user_{}", id),
            email: format!("user_{}@example.com", id),
        }
    }
}
```

## 测试质量检查清单

### 代码审查要点
- [ ] 测试是否覆盖所有业务逻辑？
- [ ] 测试是否独立可重复？
- [ ] 是否测试了边界条件？
- [ ] 是否测试了错误情况？
- [ ] Mock使用是否合理？
- [ ] 测试命名是否清晰？
- [ ] 断言消息是否明确？

### 测试执行要求
- [ ] 所有测试在提交前必须通过
- [ ] 测试执行时间不超过5分钟
- [ ] 测试覆盖率满足要求
- [ ] 集成测试覆盖所有API端点
- [ ] 性能测试覆盖关键路径

### 测试维护
- [ ] 测试代码与生产代码同步更新
- [ ] 定期清理过时的测试
- [ ] 测试数据保持最新
- [ ] 测试工具定期升级

### 测试最佳实践
1. **一个测试一个断言**：每个测试只验证一个行为
2. **测试行为而非实现**：关注what而非how
3. **使用有意义的测试数据**：避免魔法值
4. **保持测试简单**：复杂的测试难以维护
5. **测试失败信息明确**：便于快速定位问题
