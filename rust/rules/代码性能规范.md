# 代码性能规范

## 概述
本文档定义Rust项目的性能优化规范，涵盖数据库优化、缓存策略、并发控制、算法优化等方面，确保系统在高并发场景下的稳定性和响应速度。

---

## 性能原则

### 1. 测量优先
- 优化前先测量性能瓶颈
- 使用基准测试验证优化效果
- 监控生产环境性能指标

### 2. 渐进优化
- 优先优化关键路径
- 避免过早优化
- 保持代码可读性和可维护性

### 3. 资源意识
- 合理使用内存和CPU
- 避免不必要的分配和复制
- 及时释放资源

### 4. 并发友好
- 充分利用多核CPU
- 避免锁竞争
- 使用异步提高吞吐量

## 数据库性能

### 1. 查询优化
```
// 不好：SELECT * 查询
let users = sqlx::query("SELECT * FROM users")
    .fetch_all(pool)
    .await?;

// 好：只查询需要的字段
let users = sqlx::query_as!(
    UserBrief,
    "SELECT id, username, email FROM users"
)
.fetch_all(pool)
.await?;

// 好：使用分页
let users = sqlx::query_as!(
    UserBrief,
    "SELECT id, username, email FROM users LIMIT $1 OFFSET $2",
    page_size,
    (page - 1) * page_size
)
.fetch_all(pool)
.await?;
```

### 2. 索引使用
```
// 创建复合索引（示例SQL）
// CREATE INDEX idx_users_email_status ON users(email, status);

// 查询应该命中索引
let active_users = sqlx::query_as!(
    User,
    "SELECT * FROM users WHERE email LIKE $1 AND status = 'active'",
    format!("{}%", domain)
```

### 3. 批量操作
```
// 不好：循环中执行SQL（N+1问题）
for user in &users {
    sqlx::query("INSERT INTO users (username, email) VALUES ($1, $2)")
        .bind(&user.username)
        .bind(&user.email)
        .execute(pool)
        .await?;
}

// 好：批量插入
let mut query_builder: QueryBuilder<Postgres> = QueryBuilder::new(
    "INSERT INTO users (username, email) "
);

query_builder.push_values(users.iter(), |mut b, user| {
    b.push_bind(&user.username);
    b.push_bind(&user.email);
});

let query = query_builder.build();
query.execute(pool).await?;

// 好：使用事务批量操作
let mut tx = pool.begin().await?;

for user in &users {
    sqlx::query("INSERT INTO users (username, email) VALUES ($1, $2)")
        .bind(&user.username)
        .bind(&user.email)
        .execute(&mut *tx)
        .await?;
}

tx.commit().await?;
```

## 缓存性能

### 1. Redis Key设计
```
// 不好：无结构的Key
let key = format!("user_{}", user_id);

// 好：有命名空间的Key
let key = format!("app:users:{}", user_id);

// 好：带版本的Key（便于缓存清理）
let key = format!("app:v1:users:{}", user_id);

// 带过期时间的缓存
let user: Option<User> = redis::cmd("GET")
    .arg(&key)
    .query_async(&mut conn)
    .await?;

if user.is_none() {
    let user = fetch_user_from_db(user_id).await?;
    
    // 设置缓存，30分钟过期
    redis::cmd("SETEX")
        .arg(&key)
        .arg(30 * 60)  // 30分钟
        .arg(serde_json::to_string(&user)?)
        .query_async(&mut conn)
        .await?;
    
    Ok(Some(user))
} else {
    Ok(user)
}
```

### 2. Redis命令优化
```
use redis::Pipeline;

// 不好：多次单独命令
let mut conn = pool.get().await?;
let name: String = redis::cmd("GET")
    .arg("user:1:name")
    .query_async(&mut conn)
    .await?;
let email: String = redis::cmd("GET")
    .arg("user:1:email")
    .query_async(&mut conn)
    .await?;

// 好：使用管道（pipeline）

pipe.get("user:1:name")
    .get("user:1:email");

    .query_async(&mut conn)
    .await?;

// 好：使用哈希表存储相关数据
let user_data: HashMap<String, String> = redis::cmd("HGETALL")
    .arg("user:1")
    .query_async(&mut conn)
    .await?;
```

### 3. 缓存策略
```
/// 带本地缓存的Redis客户端
pub struct CachedRedisClient {
    redis_pool: RedisPool,
    local_cache: Arc<Mutex<LruCache<String, Vec<u8>>>>,
}

impl CachedRedisClient {
    pub async fn get_with_local_cache(&self, key: &str) -> Result<Option<Vec<u8>>, Error> {
        // 先检查本地缓存
        {
            let mut cache = self.local_cache.lock().await;
            if let Some(value) = cache.get(key) {
                return Ok(Some(value.clone()));
            }
        }
        
        // 从Redis获取
        let value: Option<Vec<u8>> = self.redis_pool.get(key).await?;
        
        // 更新本地缓存
        if let Some(ref value) = value {
            let mut cache = self.local_cache.lock().await;
            cache.put(key.to_string(), value.clone());
        }
        
        Ok(value)
    }
}

/// 缓存穿透防护
pub async fn get_user_with_penetration_protection(
    user_id: i64,
) -> Result<Option<User>, Error> {
    let key = format!("user:{}", user_id);
    
    // 尝试从缓存获取
    if let Some(user) = cache.get(&key).await? {
        return Ok(Some(user));
    }
    
    // 使用分布式锁防止缓存击穿
    let lock_key = format!("lock:user:{}", user_id);
    let lock = distributed_lock.acquire(&lock_key, Duration::from_secs(5)).await?;
    
    // 再次检查缓存（双重检查锁定）
    if let Some(user) = cache.get(&key).await? {
        return Ok(Some(user));
    }
    
    // 从数据库获取
    let user = user_repository.find_by_id(user_id).await?;
    
    if let Some(ref user) = user {
        // 缓存用户数据
        cache.set(&key, user, Duration::from_secs(300)).await?;
    } else {
        // 缓存空值，防止缓存穿透
        cache.set(&key, &None::<User>, Duration::from_secs(60)).await?;
    }
    
    lock.release().await?;
    
    Ok(user)
}
```

### 1. 异步编程

```
use tokio::task;

/// 并行处理多个任务
pub async fn process_users_parallel(users: Vec<User>) -> Result<Vec<ProcessedUser>, Error> {
    let mut tasks = Vec::new();
    
    for user in users {
        let task = task::spawn(async move {
            process_user(user).await
        });
        tasks.push(task);
    }
    
    let mut results = Vec::new();
    for task in tasks {
        match task.await {
            Ok(Ok(processed)) => results.push(processed),
            Ok(Err(e)) => return Err(e),
            Err(e) => return Err(Error::from(e)),
        }
    }
    
    Ok(results)
}

/// 使用select!处理多个异步操作
pub async fn fetch_user_with_timeout(
    user_id: i64,
) -> Result<User, Error> {
    tokio::select! {
        user = user_repository.find_by_id(user_id) => {
            user?.ok_or(Error::UserNotFound)
        }
        _ = tokio::time::sleep(Duration::from_secs(5)) => {
            Err(Error::Timeout)
        }
    }
}
```

### 2. 锁优化
```
use std::sync::Arc;
use tokio::sync::{RwLock, Mutex};

/// 使用读写锁提高并发读性能
pub struct UserCache {
    users: Arc<RwLock<HashMap<i64, User>>>,
}

impl UserCache {
    pub async fn get(&self, user_id: i64) -> Option<User> {
        let users = self.users.read().await;
        users.get(&user_id).cloned()
    }
    
    pub async fn update(&self, user: User) {
        let mut users = self.users.write().await;
        users.insert(user.id, user);
    }
}

/// 减少锁粒度
pub struct OptimizedCache {
    // 按用户ID分片，减少锁竞争
    shards: Vec<Arc<Mutex<HashMap<i64, User>>>>,
}

impl OptimizedCache {
    fn get_shard(&self, user_id: i64) -> &Arc<Mutex<HashMap<i64, User>>> {
        let shard_count = self.shards.len();
        let shard_index = (user_id as usize) % shard_count;
        &self.shards[shard_index]
    }
    
    pub async fn get(&self, user_id: i64) -> Option<User> {
        let shard = self.get_shard(user_id);
        let map = shard.lock().await;
        map.get(&user_id).cloned()
    }
}
```

## 代码性能优化

### 1. 避免不必要的分配
```
// 不好：频繁分配字符串
fn build_message(name: &str) -> String {
    let mut message = String::new();
    message.push_str("Hello, ");
    message.push_str(name);
    message.push_str("!");
    message
}

// 好：使用format!（一次分配）
fn build_message(name: &str) -> String {
    format!("Hello, {}!", name)
}

// 更好：返回&str避免分配
fn get_greeting() -> &'static str {
    "Hello, World!"
}

// 使用Cow避免不必要的克隆
use std::borrow::Cow;

fn process_name(name: &str) -> Cow<'_, str> {
    if name.contains(' ') {
        // 需要修改，分配新字符串
        Cow::Owned(name.replace(' ', "_"))
    } else {
        // 不需要修改，直接使用引用
        Cow::Borrowed(name)
    }
}
```

### 2. 算法优化
```
// 不好：O(n²)算法
fn find_duplicates_naive(items: &[i32]) -> Vec<i32> {
    let mut duplicates = Vec::new();
    
    for (i, &item1) in items.iter().enumerate() {
        for (j, &item2) in items.iter().enumerate() {
            if i != j && item1 == item2 && !duplicates.contains(&item1) {
                duplicates.push(item1);
            }
        }
    }
    
    duplicates
}

// 好：O(n)算法使用HashSet
use std::collections::HashSet;

fn find_duplicates_optimized(items: &[i32]) -> Vec<i32> {
    let mut seen = HashSet::new();
    let mut duplicates = HashSet::new();
    
    for &item in items {
        if !seen.insert(item) {
            duplicates.insert(item);
        }
    }
    
    duplicates.into_iter().collect()
}

// 使用迭代器链（零成本抽象）
fn process_numbers(numbers: &[i32]) -> Vec<i32> {
    numbers.iter()
        .filter(|&&n| n > 0)          // 过滤正数
        .map(|&n| n * 2)              // 乘以2
        .filter(|&n| n < 100)         // 过滤小于100的数
        .collect()                    // 收集结果
}
```

```
/// 使用缓冲写入
pub async fn write_large_file(data: &[u8], path: &str) -> Result<(), Error> {
    let file = File::create(path).await?;
    let mut writer = BufWriter::new(file);
    
    // 分批写入

        writer.write_all(chunk).await?;

    Ok(())
}

}
```

## 性能优化检查清单

### 代码审查要点
- [ ] 是否避免了不必要的内存分配？
- [ ] 算法复杂度是否合理？
- [ ] 是否使用了适当的并发原语？
- [ ] 数据库查询是否优化？
- [ ] 缓存策略是否合理？
- [ ] I/O操作是否使用缓冲？

### 性能测试要求
- [ ] 单元测试包含性能断言
- [ ] 集成测试模拟并发场景
- [ ] 基准测试覆盖关键路径
- [ ] 负载测试验证系统容量

### 生产监控指标
- [ ] 响应时间（P50, P95, P99）
- [ ] 错误率
- [ ] 吞吐量（QPS）
- [ ] 资源使用率（CPU、内存、网络）
- [ ] 缓存命中率
- [ ] 数据库连接池使用率

### 性能优化优先级
1. **关键路径优化**：影响用户体验的核心功能
2. **资源密集型操作**：消耗大量CPU、内存、I/O的操作
3. **高频调用代码**：被频繁执行的代码路径
4. **可扩展性瓶颈**：限制系统扩展的组件

### 性能工具推荐
1. **criterion**: 基准测试框架
2. **flamegraph**: 性能火焰图生成
3. **pprof**: 性能剖析工具

### 性能优化模式
1. **缓存模式**：减少重复计算
2. **池化模式**：复用昂贵资源
3. **批处理模式**：减少操作次数
4. **异步模式**：提高并发能力
5. **压缩模式**：减少数据传输量
6. **索引模式**：加速数据查找

### 性能反模式
1. **N+1查询**：循环中执行数据库查询
2. **内存泄漏**：未释放不再使用的内存
3. **阻塞操作**：在异步上下文中执行同步操作
4. **过度同步**：不必要的锁竞争
5. **过早优化**：未测量就优化
