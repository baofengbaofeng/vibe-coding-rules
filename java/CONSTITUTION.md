# CONSTITUTION.md

本宪法定义项目开发的核心原则、质量门禁和治理规则，是AI Coding和开发团队在本仓库开发的最高行为准则。

**优先级说明**：AGENTS.md > 本宪法 > rules/*.md规范 > 通用开发惯例

---

## 核心原则

### 一、分层架构原则

严格遵循分层架构，禁止越层编写代码：

| 层级 | 职责 | 禁止行为 |
|------|------|----------|
| web/controller | HTTP请求处理、参数校验、结果封装 | 禁止编写业务逻辑 |
| domain/module | 业务规则与流程编排，唯一业务逻辑存放处 | 禁止直接操作Mapper |
| persistence | 数据读写，SQL定义 | 禁止编写业务逻辑 |
| remote | 外部服务调用封装 | 禁止业务代码直接调用SDK |
| utils | 无状态通用工具 | 禁止编写业务逻辑 |

**跨层调用禁止**：Mapper代码只能在对应的Module中调用，禁止跨Module调用其他领域的Mapper。

### 二、安全优先原则

基于OWASP Top 10，安全是不可妥协的底线：

- **注入防护**：SQL必须使用`#{}`参数绑定，禁止`${}`拼接
- **敏感信息**：配置使用`ENCRYPT@`加密存储，代码禁止硬编码密钥
- **权限校验**：所有API必须经过鉴权，禁止新增无鉴权接口
- **日志脱敏**：身份证、手机号、银行卡等敏感字段必须脱敏输出

### 三、质量门禁原则

代码合入必须通过以下检查：

| 检查项 | 工具/命令 | 通过标准 | 失败处理 |
|--------|----------|----------|----------|
| 编译构建 | `mvn compile` | 无错误 | 禁止提交 |
| 代码规范 | `mvn validate` | Checkstyle无违规 | 禁止提交 |
| 单元测试 | `mvn test` | 全量通过 | 禁止提交 |
| 覆盖率 | JaCoCo | domain≥80%, web≥60% | 禁止提交 |

### 四、变更追溯原则

所有变更必须可追溯、可回滚：

- **规范驱动**：功能开发遵循 `/speckit.specify` → `/speckit.plan` → `/speckit.tasks` → `/speckit.implement` 流程
- **分支管理**：遵循 `feature/fix → dev → test → stage → master → release` 合并顺序
- **提交规范**：Commit Message遵循 `<类型>(<范围>): <描述>` 格式
- **回滚预案**：破坏性变更必须有回滚方案

### 五、第一性原理原则

以原始需求和问题本质出发，不从惯例或模板出发：

1. 不要假设用户清楚自己想要什么，动机不清晰时停下来讨论
2. 目标清晰但路径不是最短的，直接告诉用户并建议更好的办法
3. 遇到问题追根因，不打补丁，每个决策都要能回答"为什么"
4. 输出说重点，砍掉一切不改变决策的信息

## 技术约束

### 版本锁定

| 组件 | 版本 | 约束 |
|------|------|------|
| Java | 1.8 | 禁止使用JDK8以上新特性 |
| Spring Boot | 2.3.12.RELEASE | 核心框架禁止升级 |
| MyBatis | 2.3.0 | SQL写在XML中，禁止注解SQL |
| MySQL Connector | 5.1.49 | 数据库驱动禁止升级 |
| Redisson | 3.23.3 | 分布式锁必须使用Redisson |

### 依赖管理

- **禁止随意升级核心依赖**：Spring Boot、MyBatis、MySQL驱动等核心框架
- **禁止随意添加外部依赖**：新增依赖需提供选型说明并二次确认
- **定期检查漏洞**：关注CVE公告，使用`dependency-check-maven`插件

### 中间件规范

| 中间件 | 规范 |
|--------|------|
| MySQL | 禁止`SELECT *`，禁止无WHERE全表操作，索引≤5个/表 |
| Redis | Key必须加业务前缀，必须设置过期时间 |
| MQ | 消费必须幂等处理 |
| 文件存储 | 必须走项目统一封装 |

## 编码规范

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | 大驼峰 + 后缀 | `UserController`, `UserModule`, `UserEntity` |
| 方法名 | 小驼峰 + 动词前缀 | `queryUserById`, `createUser`, `updateUser` |
| 常量 | 全大写下划线 | `MAX_RETRY_COUNT` |
| 变量 | 小驼峰 | `userIdentifier`（禁止二义性缩写） |

### 格式规范

- **缩进**：4空格，禁止Tab
- **行宽**：单行不超过120字符
- **编码**：UTF-8
- **换行**：LF

### 注释规范

- **类注释**：必须Javadoc，包含`@author`、`@create`
- **公共方法**：必须Javadoc，包含`@param`、`@return`、`@throws`
- **字段注释**：必须有注释说明

### 禁止事项

1. 禁止使用`System.out`/`System.err`调试，必须使用SLF4J
2. 禁止捕获裸`Exception`/`Throwable`，必须捕获具体类型
3. 禁止星号导入，每行导入一个类
4. 禁止魔法数字，必须定义为常量
5. 禁止使用`java.util.Date`，必须使用`java.time`

## 测试规范

### 覆盖率要求

| 代码类型 | 行覆盖率 | 分支覆盖率 |
|---------|---------|-----------|
| domain层 | ≥80% | ≥70% |
| web层 | ≥60% | ≥50% |
| utils层 | ≥70% | ≥60% |
| entity层 | ≥50% | - |

### 测试编写

- **隔离依赖**：禁止`@SpringBootTest`，必须Mock外部依赖
- **明确断言**：每个测试方法必须有断言
- **命名规范**：`shouldXXXWhenYYY` 或 `isXXXWhenYYY`
- **结构模式**：Given-When-Then

### 技术栈

- JUnit 5 (Jupiter)
- Mockito 3.12.4
- AssertJ 3.20.2
- JaCoCo

## 性能规范

### 数据库性能

- **禁止全表扫描**：WHERE条件必须命中索引
- **禁止循环SQL**：必须使用批量操作
- **批量限制**：单次批量操作≤1000条

### 缓存性能

- **Key命名**：必须加业务前缀，如`project:user:1001`
- **过期时间**：必须设置，避免内存泄漏
- **禁止KEYS命令**：生产环境使用SCAN

### 并发控制

- **禁止直接创建线程**：必须使用线程池
- **分布式锁**：必须使用Redisson，必须设置超时
- **ThreadLocal**：必须在finally中调用remove()

## 安全规范

### 注入防护

| 类型 | 防护措施 |
|------|----------|
| SQL注入 | 使用`#{}`参数绑定，禁止`${}`拼接 |
| 命令注入 | 禁止Runtime.exec拼接用户输入 |
| XPath注入 | 使用变量解析，禁止字符串拼接 |

### 敏感信息

- **配置加密**：数据库密码、Redis密码、Token等使用`ENCRYPT@`加密
- **字段加密**：身份证、银行卡、手机号使用`@PersistenceCryptoField`加密存储
- **日志脱敏**：敏感字段必须脱敏后输出

### 访问控制

- **服务端API**：必须经过`RequestAccessInterceptor`验证
- **前端API**：必须经过`RequestPermitInterceptor`验证
- **数据权限**：业务操作必须校验数据权限维度

## 提交规范

### 提交前检查

```
# 完整检查（推荐）
mvn clean validate test

# 快速检查
mvn validate
mvn test
```

### Commit Message格式

```
<类型>(<范围>): <描述>

[可选：详细说明]

[可选：关联信息]
```

**类型**：`feat` | `fix` | `test` | `docs` | `refactor` | `perf` | `style` | `chore`

### 分支管理

```
feature/20260515_功能描述  → dev → test → stage → master → release
fix/20260515_问题描述      → dev → test → stage → master → release
```

### PR要求

- 至少1位Reviewer审批
- CI流水线全部通过
- 无未解决冲突
- 合入前squash/rebase

## 红线行为

以下行为严格禁止，违者需承担相应责任：

### 代码红线

- 禁止在代码中硬编码密钥、密码、内部地址
- 禁止提交`System.out`、`e.printStackTrace()`调试代码
- 禁止提交注释掉的无效代码
- 禁止编写越层、强耦合、无注释的冗余代码

### 流程红线

- 禁止跳过单元测试或Checkstyle提交代码
- 禁止`git push --force`到共享分支
- 禁止直接合入master，必须通过PR流程
- 禁止无回滚方案的破坏性变更

### 架构红线

- 禁止修改项目核心架构与目录结构（需先评审）
- 禁止随意升级核心框架版本
- 禁止跨Module调用其他领域的Mapper

## 治理机制

### 宪法修订

- 修订需有充分理由并记录在案
- 修订后需同步更新相关文档
- 重大修订需团队评审通过

### 合规验证

- 所有PR必须验证合规性
- 发现违规必须立即修复
- 定期进行合规审查

### 问题上报

- 发现安全漏洞立即上报
- 发现架构违规立即上报
- 发现敏感信息泄露立即上报
