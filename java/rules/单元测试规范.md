# 单元测试规范.md

本规范定义项目单元测试的编写标准，确保测试代码质量、可维护性和覆盖率要求。

---

## 技术栈

- **测试框架**: JUnit 5 (Jupiter)
- **Mock框架**: Mockito 3.12.4 + Mockito Inline（支持静态方法Mock）
- **断言库**: AssertJ 3.20.2
- **覆盖率工具**: JaCoCo

---

## 测试文件组织

### 目录结构

```
src/test/java/com/yourcompany/project/test/
├── common/                     # 通用类测试
├── configuration/              # 配置类测试
├── domain/                     # 领域层测试
│   ├── subdomain1/             # 业务领域1测试
│   ├── subdomain2/             # 业务领域2测试
│   ├── subdomain3/             # 业务领域3测试
├── persistence/                # 持久化层测试
│   ├── entity/                 # 实体类测试
│   ├── enums/                  # 枚举类测试
│   └── mapper/                 # Mapper测试
├── remote/                     # 远程调用测试
├── schedule/                   # 定时任务测试
├── security/                   # 安全相关测试
├── utils/                      # 工具类测试
└── web/                        # Web层测试
    ├── controllers/            # 控制器测试
    └── context/                # 上下文测试
```

### 命名规范

- **测试类命名**: 被测类名 + `Test`，如 `UserProfileModuleTest`
- **测试方法命名**: 使用小驼峰命名，方法名描述测试场景，如：
  - `shouldReturnUserProfileWhenExists`（正向场景）
  - `shouldThrowExceptionWhenUserProfleNotFound`（异常场景）
  - `isAccessibleWhenUserProfileStatusIsActive`（条件场景）

## 编写规范

### 基础约束

1. **必须隔离外部依赖**：测试不得依赖数据库、Redis、外部服务等，必须使用Mock
2. **必须包含明确断言**：每个测试方法必须有至少一个断言
3. **禁止使用 `@SpringBootTest`**：单元测试应轻量化，禁止启动Spring容器
4. **禁止测试私有方法**：只测试公共方法，私有方法通过公共方法间接测试
5. **单测必须可重复执行**：测试之间不能有依赖关系，执行顺序不影响结果

### 测试方法结构

遵循 **Given-When-Then** 模式：

```
@Test
void shouldEncryptSuccessfully() throws Exception {
    // Given - 准备测试数据和Mock
    CryptoService cryptoService = mock(CryptoService.class);
    when(cryptoService.encrypt(anyString())).thenReturn("encryptedValue");
    MyUtil util = new MyUtil(cryptoService);

    // When - 执行被测方法
    String result = util.process("testValue");

    // Then - 验证结果
    assertThat(result).isEqualTo("encryptedValue");
}
```

### 注解使用

```
import org.junit.jupiter.api.Test;                    // 测试方法
import org.junit.jupiter.api.BeforeEach;              // 每个测试方法前执行
import org.junit.jupiter.api.AfterEach;               // 每个测试方法后执行
import org.junit.jupiter.api.BeforeAll;               // 类级别初始化（静态方法）
import org.junit.jupiter.api.DisplayName;             // 测试显示名称
import org.junit.jupiter.api.Nested;                  // 嵌套测试类
import org.junit.jupiter.params.ParameterizedTest;    // 参数化测试
import org.junit.jupiter.params.provider.ValueSource; // 参数来源
```

## 分层测试规范

### Controller层测试

Controller测试应验证HTTP请求处理、参数校验、结果封装，禁止验证业务逻辑。

```
class UserControllerTest {

    @Test
    void shouldReturnUserDetailsWhenUserExists() {
        // Given
        UserService userService = mock(UserService.class);
        UserResponse userResponse = new UserResponse();
        userResponse.setUserId(1L);
        userResponse.setUserName("testUser");
        when(userService.getUserById(1L)).thenReturn(userResponse);

        UserController controller = new UserController(userService);

        MockHttpServletRequest request = new MockHttpServletRequest();
        MockHttpServletResponse response = new MockHttpServletResponse();

        // When
        try (MockedStatic<RequestContext> mockedStatic = Mockito.mockStatic(RequestContext.class)) {
            mockedStatic.when(RequestContext::getCurrentUser).thenReturn("admin");
            controller.getUser(request, response, 1L);
        }

        // Then
        assertThat(response.getStatus()).isEqualTo(200);
    }
}
```

**要点**：
- Mock所有依赖的Module和Service
- 使用 `MockHttpServletRequest` 和 `MockHttpServletResponse`
- 静态方法使用 `MockedStatic` 进行Mock
- 只验证HTTP层面的行为，不验证业务规则

### Module层测试

Module测试验证业务逻辑，这是单元测试的核心区域。

```
class UserModuleTest {

    private UserMapper userMapper;
    private UserModule userModule;

    @BeforeEach
    void setUp() {
        userMapper = mock(UserMapper.class);
        userModule = new UserModule(userMapper);
    }

    @Test
    void shouldReturnUserWhenExists() {
        // Given
        UserEntity entity = new UserEntity();
        entity.setId(1L);
        entity.setUserName("testUser");
        when(userMapper.selectById(1L)).thenReturn(entity);

        // When
        UserEntity result = userModule.queryUserById(1L);

        // Then
        assertThat(result).isNotNull();
        assertThat(result.getId()).isEqualTo(1L);
        assertThat(result.getUserName()).isEqualTo("testUser");
    }

    @Test
    void shouldThrowExceptionWhenUserNotFound() {
        // Given
        when(userMapper.selectById(1L)).thenReturn(null);

        // When & Then
        assertThatThrownBy(() -> userModule.queryUserById(1L))
                .isInstanceOf(BusinessException.class)
                .hasMessage("用户不存在");
    }
}
```

**要点**：
- 使用 `@BeforeEach` 初始化测试环境
- Mock所有外部依赖（Mapper、远程服务等）
- 验证业务规则的正确性
- 覆盖正常流程和异常流程

### Entity层测试

Entity测试验证实体类的属性访问和业务语义方法。

```
class UserEntityTest {

    @Test
    void isActiveWhenStatusIsOne() {
        UserEntity entity = new UserEntity();
        entity.setStatus(1);
        assertTrue(entity.isActive());
    }

    @Test
    void notActiveWhenStatusIsNotOne() {
        UserEntity entity = new UserEntity();
        entity.setStatus(0);
        assertFalse(entity.isActive());
    }

    @Test
    void builderShouldWork() {
        UserEntity entity = UserEntity.builder()
                .id(1L)
                .userName("testUser")
                .status(1)
                .build();
        assertEquals(1L, entity.getId());
        assertEquals("testUser", entity.getUserName());
        assertEquals(1, entity.getStatus());
    }

    @Test
    void defaultConstructorShouldCreateInstance() {
        UserEntity entity = new UserEntity();
        assertNotNull(entity);
        assertNull(entity.getId());
        assertNull(entity.getUserName());
    }
}
```

**要点**：
- 测试Builder模式的正确性
- 测试业务语义方法（如 `isActive`）
- 测试边界条件（如null值处理）

### Utils层测试

工具类测试应覆盖各种边界情况和异常处理。

```
class CryptoUtilTest {

    @Test
    void shouldEncryptSuccessfully() throws Exception {
        CryptoService cryptoService = mock(CryptoService.class);
        when(cryptoService.encrypt(anyString())).thenReturn("encryptedValue");

        CryptoUtil util = new CryptoUtil(cryptoService);
        ReflectionTestUtils.setField(util, "algorithm", "AES");

        String result = util.encrypt("testValue");
        assertThat(result).isEqualTo("encryptedValue");
    }

    @Test
    void shouldThrowExceptionWhenEncryptFails() throws Exception {
        CryptoService cryptoService = mock(CryptoService.class);
        when(cryptoService.encrypt(anyString()))
                .thenThrow(new RuntimeException("encrypt error"));

        CryptoUtil util = new CryptoUtil(cryptoService);

        assertThatThrownBy(() -> util.encrypt("testValue"))
                .isInstanceOf(BusinessException.class);
    }

    @Test
    void shouldReturnEmptyWhenInputIsNull() {
        CryptoUtil util = new CryptoUtil(mock(CryptoService.class));

        String result = util.encrypt(null);
        assertThat(result).isEmpty();
    }
}
```

**要点**：
- 测试各种输入参数组合
- 测试异常场景
- 使用 `ReflectionTestUtils` 设置私有字段

## Mock规范

### Mockito使用

```
import static org.mockito.Mockito.mock;           // 创建Mock对象
import static org.mockito.Mockito.when;           // 定义Mock行为
import static org.mockito.Mockito.verify;         // 验证方法调用
import static org.mockito.Mockito.times;          // 验证调用次数
import static org.mockito.Mockito.any;            // 任意参数匹配
import static org.mockito.Mockito.anyString;      // 任意字符串参数
import static org.mockito.Mockito.anyLong;        // 任意Long参数
import static org.mockito.Mockito.eq;             // 精确参数匹配
```

### 静态方法Mock

使用 `Mockito.mockStatic()` Mock静态方法，必须在try-with-resources中使用：

```
try (MockedStatic<SomeStaticClass> mockedStatic = Mockito.mockStatic(SomeStaticClass.class)) {
    mockedStatic.when(SomeStaticClass::staticMethod).thenReturn("mockedValue");
    // 执行测试
}
```

### 参数匹配器

```
// 任意参数
when(mapper.selectById(anyLong())).thenReturn(entity);

// 精确匹配
when(mapper.selectById(eq(1L))).thenReturn(entity);

// 组合使用
when(mapper.selectByCondition(anyString(), anyInt(), eq("status")))
    .thenReturn(entityList);
```

### 验证调用

```
// 验证方法被调用一次
verify(mapper).selectById(1L);

// 验证方法被调用指定次数
verify(mapper, times(2)).selectById(1L);

// 验证方法从未被调用
verify(mapper, never()).deleteById(anyLong());
```

## 断言规范

### AssertJ推荐用法

```
import static org.assertj.core.api.Assertions.assertThat;
import static org.assertj.core.api.Assertions.assertThatThrownBy;

// 基本断言
assertThat(result).isNotNull();
assertThat(result).isEqualTo(expected);
assertThat(result).isTrue();

// 集合断言
assertThat(list).hasSize(3);
assertThat(list).containsExactly("a", "b", "c");
assertThat(list).doesNotContainNull();

// 异常断言
assertThatThrownBy(() -> service.doSomething())
    .isInstanceOf(BusinessException.class)
    .hasMessage("error message");

// Optional断言
assertThat(optional).isPresent();
assertThat(optional).contains(expected);

// 字符串断言
assertThat(text).startsWith("prefix");
assertThat(text).endsWith("suffix");
assertThat(text).contains("substring");
```

### JUnit断言（用于简单场景）

```
import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.junit.jupiter.api.Assertions.assertTrue;
import static org.junit.jupiter.api.Assertions.assertFalse;
import static org.junit.jupiter.api.Assertions.assertNotNull;
import static org.junit.jupiter.api.Assertions.assertThrows;

assertEquals(expected, actual);
assertTrue(condition);
assertThrows(ExpectedException.class, () -> method());
```

## 测试覆盖要求

### 必须覆盖的场景

1. **正常流程**：主要业务逻辑的正常执行路径
2. **异常流程**：参数校验失败、业务异常、系统异常
3. **边界条件**：空值、空集合、最大值、最小值
4. **分支逻辑**：if-else各分支、switch各case

### 参数化测试

对于需要多组输入的测试，使用参数化测试：

```
@ParameterizedTest
@ValueSource(strings = {"AES", "DES", "RSA"})
void shouldSupportMultipleAlgorithms(String algorithm) {
    // 测试逻辑
}

@ParameterizedTest
@CsvSource({"1, true", "0, false", "-1, false"})
void shouldValidateStatus(int status, boolean expected) {
    UserEntity entity = new UserEntity();
    entity.setStatus(status);
    assertThat(entity.isActive()).isEqualTo(expected);
}

@ParameterizedTest
@MethodSource("provideTestCases")
void shouldHandleMultipleCases(String input, String expected) {
    // 测试逻辑
}

static Stream<Arguments> provideTestCases() {
    return Stream.of(
        Arguments.of("input1", "expected1"),
        Arguments.of("input2", "expected2")
    );
}
```

## 命名与注释

### 测试方法命名

测试方法名应清晰描述测试场景：

```
// 推荐：描述期望行为
void shouldReturnUserWhenExists()
void shouldThrowExceptionWhenUserNotFound()

// 推荐：描述条件判断
void isActiveWhenStatusIsOne()
void notActiveWhenStatusIsZero()

// 不推荐：模糊命名
void test1()
void testUser()
```

### 测试类注释

测试类应添加简要注释说明测试范围：

```
/**
 * 用户模块业务逻辑测试。
 */
class UserModuleTest {
    // ...
}
```

## 最佳实践

1. **单一职责**：每个测试方法只验证一个行为
2. **快速执行**：单元测试应在秒级完成
3. **可读性**：测试代码应清晰表达测试意图
4. **可维护性**：测试代码应与业务代码同步更新
5. **隔离性**：测试之间完全独立，不共享状态

---

## 禁止事项

1. **禁止使用 `@SpringBootTest`**：单元测试不应启动Spring容器
2. **禁止依赖外部系统**：数据库、Redis、HTTP服务等必须Mock
3. **禁止使用 `System.out` 打印**：测试结果通过断言验证
4. **禁止无断言的测试**：每个测试必须有明确的验证点
5. **禁止测试间依赖**：每个测试方法必须独立可执行
6. **禁止捕获裸 `Exception`**：必须捕获具体异常类型
