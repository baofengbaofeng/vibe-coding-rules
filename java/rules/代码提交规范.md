# 代码提交规范.md

本规范定义代码提交的标准流程、Commit Message格式、分支管理策略以及提交前检查要求，确保代码仓库的整洁与可追溯性。

---

## 提交前检查单

**每次提交前必须确保以下检查全部通过**：

### 单元测试
```
mvn test
```
要求：全量通过，无失败用例

### 代码规范
```
mvn validate
```
要求：Checkstyle检查通过，无违规

### 编译构建
```
mvn compile
```
要求：编译成功，无错误

### 快速检查命令

```
# 执行完整检查（推荐）
mvn clean validate test

# 仅检查代码规范
mvn validate

# 仅执行单元测试
mvn test

# 跳过测试的本地打包（仅用于快速验证编译）
mvn clean package -Dmaven.test.skip=true
```

### 检查失败处理

- **单元测试失败**：必须修复失败用例，禁止跳过测试提交
- **Checkstyle违规**：根据错误提示修复代码风格问题
- **编译失败**：解决编译错误后重新提交

## 提交消息规范

### 消息格式

```
<提交类型>(<关联范围>): <提交标题>

<消息内容>

<关联信息>
```

### 提交类型

- 功能迭代：`feat`，如：`feat(user): 新增用户注册功能`
- 测试相关：`test`，如：`test(user): 用户模块单元测试`
- 文档相关：`docs`，如：`docs: 更新API接口文档`
- 性能优化：`perf`，如：`perf(query): 优化批量查询性能`
- 代码重构：`refactor`，如：`refactor(user): 重构用户查询逻辑`
- 代码格式：`style`，如：`style: 统一缩进为4空格`
- 构建相关：`chore`，如：`chore: 升级Maven插件版本`
- 问题修复：`fix`，如：`fix(auth): 修复登录状态丢失问题`

### 关联范围

根据修改的模块或层级填写：

- `controller`、`web` - Web层
- `domain` - 业务领域层
- `persistence` - 数据持久化层
- `remote` - 远程调用
- `config` - 配置相关
- `security` - 安全相关
- `schedule` - 定时任务
- `utils` - 工具类
- `enhance` - 组件优化

### 提交标题

- 使用祈使句，首字母小写
- 不超过50个字符
- 结尾不加句号
- 简明扼要描述变更内容

### 消息内容

- 详细描述变更原因、影响范围
- 每行不超过72个字符
- 使用列表形式说明多个变更点

### 关联信息（可选）

- 关联Issue：`Closes #123` 或 `Fixes #456`
- 破坏性变更：`BREAKING CHANGE: 描述变更影响`

### 示例

```
feat(module): 新增用户档案批量导出功能

- 支持按条件筛选导出范围
- 支持异步导出，避免长连接超时
- 导出结果通过消息队列通知

Closes #234
```

```
fix(mapper): 修复查询条件空指针异常

当查询参数为空时，MyBatis动态SQL未正确处理，
导致NPE异常。增加空值判断逻辑。

Fixes #567
```

## 分支管理规范

### 分支类型

- 生产分支：`master`，用于进行生产环境部署
- 预发分支：`stage` 或 `stage-xxx`, 用于进行预发环境部署
- 测试分支：`test`，用于进行测试环境部署
- 功能分支：`feature/*`，功能开发分支，命名：`feature/日期_功能描述`
- 历史分支：`release/*`，历史版本分支，命名：`release/日期`
- 修复分支：`fix/*`，用于修复代码错误的分支，命名：`fix/日期_问题描述`
- 开发分支：`dev`，用于开发环境部署、集成测试

### 分支操作流程

#### 功能开发

```
# 1. 从dev创建功能分支
git checkout dev
git pull origin dev
git checkout -b feature/20260515_功能描述

# 2. 开发并提交
git add .
git commit -m "feat(module): 功能描述"

# 3. 推送到远程
git push origin feature/20260515_功能描述

# 4. 创建PR合入dev分支
```

#### 问题修复

```
# 1. 从dev创建修复分支
git checkout dev
git pull origin dev
git checkout -b fix/20260515_问题描述

# 2. 修复并提交
git add .
git commit -m "fix(module): 修复描述"

# 3. 推送并创建PR
git push origin fix/20260515_问题描述
```

### 合并流程

所有的代码合并都必须满足 `feature/fix → dev → test → stage → master → release` 的顺序要求，禁止在dev、test、stage、master、release 分支直接提交代码变更

## Pull Request规范

### PR标题

与提交消息格式一致：`<提交类型>(<关联范围>): <提交标题>`

### PR模版

```
## 变更说明
简要描述本次变更的内容和目的。

## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 性能优化
- [ ] 文档更新

## 测试情况
- [ ] 单元测试已通过
- [ ] 新增测试用例覆盖变更代码
- [ ] 本地功能验证通过

## 影响范围
列出可能受影响的模块或接口。

## 检查清单
- [ ] 代码符合编码规范
- [ ] 无敏感信息泄露
- [ ] 向下兼容，无破坏性变更
```

### PR合入要求

- 至少1位Reviewer审批通过
- CI流水线全部通过（编译、CheckStyle、单元测试）
- 无未解决的冲突
- 合入前squash或rebase保持提交历史整洁

## 提交禁止事项

### 禁止提交的内容

1. **敏感信息**：密钥、密码、内部地址、Token等
2. **临时文件**：IDE配置、日志文件、临时目录
3. **注释代码**：已废弃的注释掉的代码块
4. **调试代码**：`System.out`、`System.err`、`e.printStackTrace()`等调试语句
5. **二进制文件**：大型图片、压缩包等非必要二进制文件

### 禁止的操作

1. **禁止强制推送**：`git push --force` 到共享分支
2. **禁止跳过检查提交**：绕过单元测试或Checkstyle
3. **禁止直接合入master**：必须通过PR流程
4. **禁止无回滚方案**：破坏性变更必须有回滚预案
5. **禁止大体积提交**：单次提交变更不超过500行（重构除外）

## 提交最佳实践

### 提交粒度

- **原子性**：每次提交应是一个完整的逻辑单元
- **独立性**：每次提交应可独立回滚
- **可读性**：Commit Message应清晰表达变更意图

### 提交频率

- 完成一个功能点或修复一个问题后立即提交
- 避免积累过多变更后一次性提交
- 工作结束前确保所有变更已提交

### 代码审查

- 提交前自行Review变更内容
- 关注是否有遗漏的调试代码
- 确认无敏感信息泄露

## 常见问题处理

### 撤销未推送的提交

```
# 撤销最近一次提交，保留变更
git reset --soft HEAD~1

# 撤销最近一次提交，丢弃变更
git reset --hard HEAD~1
```

### 修改最近一次提交信息

```
git commit --amend -m "新的提交信息"
```

### 解决合并冲突

```
# 1. 拉取最新代码
git fetch origin

# 2. 变基
git rebase origin/dev

# 3. 解决冲突后继续
git add .
git rebase --continue
```
