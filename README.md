# Vibe Coding Rules

一个多语言、企业级的编码规范与开发标准仓库，为AI辅助开发（Claude Code）和开发团队提供统一的编码标准、架构规范和最佳实践。

## 📋 项目概述

Vibe Coding Rules 是一个系统化的编码规范仓库，旨在为不同技术栈的项目提供：
- **统一的编码标准**：跨语言的一致性编码规范
- **企业级架构规范**：分层架构、安全规范、性能优化
- **AI辅助开发优化**：为Claude Code等AI开发工具提供结构化指导
- **质量门禁体系**：代码审查、测试覆盖、安全扫描的完整流程

## 🏗️ 架构设计

### 核心原则
1. **分层架构原则**：严格遵循分层架构，禁止越层编写代码
2. **安全优先原则**：基于OWASP Top 10，安全是不可妥协的底线
3. **质量门禁原则**：代码合入必须通过语法检查、代码规范、类型检查、单元测试、覆盖率检查
4. **变更追溯原则**：所有变更必须可追溯、可回滚
5. **第一性原理原则**：以原始需求和问题本质出发，不从惯例或模板出发

### 规范优先级
```
AGENTS.md > CONSTITUTION.md > rules/*.md规范 > 通用开发惯例
```

## 📚 核心规范

### 1. 编码规范
- **命名规范**：类名大驼峰、方法名小驼峰、常量全大写
- **格式规范**：4空格缩进、120字符行宽、UTF-8编码、LF换行
- **注释规范**：类必须Docstring、公共方法必须Docstring、字段必须有注释
- **禁止事项**：禁止print()调试、禁止捕获裸Exception、禁止魔法数字

### 2. 安全规范
- **注入防护**：SQL参数绑定、命令注入防护、XPath注入防护
- **敏感信息**：配置加密存储、字段加密存储、日志脱敏
- **访问控制**：API鉴权、权限校验、操作审计
- **安全配置**：CORS白名单、安全响应头、速率限制

### 3. 性能规范
- **数据库性能**：禁止全表扫描、禁止循环SQL、批量操作限制
- **缓存性能**：Key业务前缀、设置过期时间、禁止KEYS命令
- **并发控制**：异步编程、分布式锁、线程池管理

### 4. 测试规范
- **覆盖率要求**：domain层≥80%、web层≥60%、utils层≥70%
- **测试编写**：隔离依赖、明确断言、命名规范、结构模式
- **技术栈**：pytest + pytest-mock + pytest-cov + pytest-tornado

### 5. 提交规范
- **提交前检查**：语法检查、代码规范、类型检查、单元测试
- **Commit Message**：`<类型>(<范围>): <描述>`格式
- **分支管理**：feature/fix → dev → test → stage → master → release
- **PR要求**：至少1位Reviewer、CI流水线通过、无未解决冲突

## 🔧 使用方式

### 对于开发团队
1. **初始化项目**：将对应语言目录的规范文件复制到项目文档目录
2. **集成到CI/CD**：将规范检查集成到流水线中
3. **代码审查**：使用规范作为代码审查的标准
4. **新人培训**：作为新成员的技术培训材料

### 对于AI辅助开发（Claude Code）
1. **配置Claude Code**：在项目根目录放置CLAUDE.md指向AGENTS.md
2. **自动规范加载**：细分规范通过glob路径自动按需加载
3. **技能调用**：使用预定义的技能进行代码检查
4. **记忆系统**：利用MEMORY.md记录项目历史决策和业务背景

### 对于项目维护者
1. **规范更新**：根据技术演进和团队反馈更新规范
2. **版本管理**：规范版本化，记录变更历史
3. **合规验证**：定期进行合规审查，发现违规立即修复
4. **问题上报**：发现安全漏洞、架构违规立即上报

## 🚨 红线行为

### 代码红线
- 禁止在代码中硬编码密钥、密码、内部地址
- 禁止提交`print()`调试代码
- 禁止提交注释掉的无效代码
- 禁止编写越层、强耦合、无注释的冗余代码

### 流程红线
- 禁止跳过单元测试或代码规范检查提交代码
- 禁止`git push --force`到共享分支
- 禁止直接合入master，必须通过PR流程
- 禁止无回滚方案的破坏性变更

### 架构红线
- 禁止修改项目核心架构与目录结构（需先评审）
- 禁止随意升级核心框架版本
- 禁止跨领域调用其他领域的Repository/DAO

## 🤝 贡献指南

### 如何贡献
1. **发现问题**：在实际使用中发现规范不完善或错误
2. **提出建议**：通过Issue或PR提出改进建议
3. **提交PR**：按照提交规范提交修改
4. **讨论评审**：参与规范讨论和代码评审

### 贡献原则
1. **一致性**：保持跨语言规范的一致性
2. **实用性**：规范应该解决实际问题
3. **可操作性**：规范应该易于执行和检查
4. **演进性**：规范应该随着技术发展而演进
