## 用户输入

```
$ARGUMENTS
```

## 目标

执行Python项目代码性能检查，包括：
1. 数据库性能检查（SQL优化、索引使用、批量操作）
2. 缓存性能检查（Redis Key设计、过期时间、命令使用）
3. 并发控制检查（异步编程、分布式锁、线程池）
4. 代码性能检查（循环优化、算法复杂度、内存使用）
5. 输出详细的性能检查报告

## 执行步骤

### 1. 确认项目环境

首先确认当前目录是Python项目根目录，检查以下技术栈：
- Python 3.12+
- SQLAlchemy 2.0+（数据库ORM）
- Redis + redis-py 5.0+（缓存）
- Tornado 6.3+（异步Web框架）

### 2. 数据库性能检查

#### SQL优化检查
- 检查是否使用 `SELECT *`（应指定具体字段）
- 检查WHERE条件是否命中索引
- 检查是否进行全表扫描
- 检查JOIN操作是否合理

#### 索引使用检查
- 检查频繁查询字段是否建立索引
- 检查索引数量是否合理（≤5个/表）
- 检查复合索引字段顺序

#### 批量操作检查
- 检查是否在循环中执行SQL（N+1查询问题）
- 检查是否使用批量操作替代循环操作
- 检查批量操作大小限制（≤1000条）

#### 事务使用检查
- 检查事务范围是否合理
- 检查长事务问题
- 检查事务隔离级别

### 3. 缓存性能检查

#### Redis Key设计检查
- 检查Key是否加业务前缀（如 `project:user:1001`）
- 检查Key命名是否规范
- 检查Key是否设置过期时间

#### Redis命令使用检查
- 检查是否在生产环境使用 `KEYS` 命令（应使用 `SCAN`）
- 检查是否合理使用管道（pipeline）
- 检查是否使用合适的数据结构

#### 缓存策略检查
- 检查缓存穿透防护（空值缓存）
- 检查缓存雪崩防护（随机过期时间）
- 检查缓存击穿防护（分布式锁）

### 4. 并发控制检查

#### 异步编程检查
- 检查是否使用Tornado的异步特性
- 检查是否阻塞IOLoop（如同步数据库操作）
- 检查协程使用是否合理

#### 分布式锁检查
- 检查分布式锁是否使用redis-py实现
- 检查锁是否设置超时时间
- 检查锁释放逻辑是否正确

#### 线程池检查
- 检查是否直接创建线程（应使用线程池）
- 检查线程池配置是否合理
- 检查线程安全代码

### 5. 代码性能检查

#### 循环优化检查
- 检查循环中是否包含重复计算
- 检查是否可以使用列表推导式
- 检查算法时间复杂度

#### 内存使用检查
- 检查大对象内存占用
- 检查内存泄漏风险
- 检查生成器使用

#### I/O操作检查
- 检查文件操作是否使用缓冲
- 检查网络请求是否使用连接池
- 检查序列化/反序列化性能

## 检查规则参考

### 数据库性能规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| SELECT * | 指定具体字段 | LOW |
| N+1查询 | 使用批量查询或JOIN | HIGH |
| 全表扫描 | WHERE条件命中索引 | HIGH |
| 批量操作 | 单次≤1000条 | MEDIUM |

### 缓存性能规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| KEYS命令 | 使用SCAN替代 | HIGH |
| Key过期时间 | 必须设置 | MEDIUM |
| 缓存穿透 | 空值缓存 | MEDIUM |

### 并发控制规则

| 检查项 | 性能要求 | 风险等级 |
|--------|----------|----------|
| 阻塞IOLoop | 使用异步操作 | HIGH |
| 分布式锁超时 | 必须设置 | MEDIUM |
| 线程池配置 | 合理配置 | MEDIUM |

## 注意事项

1. 性能检查需要结合具体业务场景和数据量
2. 性能优化应遵循"先测量，后优化"原则
3. 建议在性能测试环境执行检查
4. 性能优化可能涉及架构调整，需评审
5. 遵循AGENTS.md中的分层架构原则
