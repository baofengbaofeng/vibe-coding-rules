## 用户输入

```
$ARGUMENTS
```

## 目标

执行Python项目单元测试检查，包括：
1. 测试覆盖率检查（行覆盖率、分支覆盖率）
2. 测试质量检查（测试隔离、明确断言、命名规范）
3. 测试规范检查（测试结构、技术栈、最佳实践）
4. 测试完整性检查（核心业务测试覆盖）
5. 输出详细的测试检查报告

## 执行步骤

### 1. 确认项目环境

首先确认当前目录是Python项目根目录，检查以下测试技术栈：
- pytest 7.4+
- pytest-mock 3.12+
- pytest-cov 4.1+
- pytest-tornado 2.0+（Tornado项目）

### 2. 测试覆盖率检查

#### 行覆盖率检查
- 检查domain层行覆盖率是否≥80%
- 检查handlers层行覆盖率是否≥60%
- 检查utils层行覆盖率是否≥70%
- 检查整体行覆盖率

#### 分支覆盖率检查
- 检查domain层分支覆盖率是否≥70%
- 检查handlers层分支覆盖率是否≥50%
- 检查utils层分支覆盖率是否≥60%

#### 覆盖率详情分析
- 分析未覆盖的代码行
- 识别关键业务逻辑未覆盖
- 检查测试死角

### 3. 测试质量检查

#### 测试隔离检查
- 检查是否启动完整应用（应Mock外部依赖）
- 检查测试是否依赖外部服务（数据库、Redis、第三方API）
- 检查测试是否可重复执行

#### 明确断言检查
- 检查每个测试方法是否有断言
- 检查断言是否具体明确
- 检查异常测试是否有断言

#### 命名规范检查
- 检查测试方法命名是否符合规范（`test_should_xxx_when_yyy`）
- 检查测试文件命名是否符合规范（`test_*.py` 或 `*_test.py`）

#### 测试结构检查
- 检查是否遵循Given-When-Then结构
- 检查测试准备（setup）和清理（teardown）是否合理
- 检查测试数据管理

### 4. 测试规范检查

#### 技术栈使用检查
- 检查是否使用pytest而非unittest
- 检查是否使用pytest-mock进行Mock
- 检查是否使用pytest-cov生成覆盖率
- 检查Tornado项目是否使用pytest-tornado

#### 最佳实践检查
- 检查是否测试边界条件
- 检查是否测试异常情况
- 检查是否测试性能关键路径
- 检查测试是否独立不相互依赖

### 5. 测试完整性检查

#### 核心业务测试检查
- 检查domain层核心业务逻辑是否都有测试
- 检查handlers层关键接口是否都有测试
- 检查utils层重要工具函数是否都有测试

#### 新代码测试检查
- 检查新增代码是否都有对应测试
- 检查修改代码是否更新了相关测试
- 检查删除代码是否清理了相关测试

## 检查规则参考

### 覆盖率要求规则

| 代码类型 | 行覆盖率要求 | 分支覆盖率要求 |
|---------|-------------|---------------|
| domain层 | ≥80% | ≥70% |
| handlers层 | ≥60% | ≥50% |
| utils层 | ≥70% | ≥60% |
| 整体 | ≥70% | ≥60% |

### 测试质量规则

| 检查项 | 质量要求 | 严重级别 |
|--------|----------|----------|
| 测试隔离 | Mock外部依赖，不启动完整应用 | HIGH |
| 明确断言 | 每个测试方法必须有断言 | HIGH |
| 命名规范 | `test_should_xxx_when_yyy` 格式 | MEDIUM |
| 测试结构 | Given-When-Then结构 | MEDIUM |

### 测试规范规则

| 检查项 | 规范要求 | 严重级别 |
|--------|----------|----------|
| 技术栈 | 使用pytest + pytest-mock + pytest-cov | MEDIUM |
| 边界测试 | 测试边界条件和异常情况 | MEDIUM |
| 核心业务 | 核心业务逻辑必须有测试 | HIGH |

## 使用示例

```
# 全项目测试检查
/code-tests-check

# 检查指定模块测试
/code-tests-check domain

# 生成详细覆盖率报告
/code-tests-check --cov-report=html
```

## 注意事项

1. 测试检查需要先运行测试生成覆盖率数据
2. 测试质量比测试数量更重要
3. 测试应该随着代码演进而更新
4. 测试失败应该立即修复
5. 遵循AGENTS.md中的质量门禁原则
